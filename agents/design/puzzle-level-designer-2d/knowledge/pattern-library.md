# Pattern Library (Universal)

Patterns stop random obstacle placement by providing named reusable templates. Every pattern includes purpose, ingredients, common mistakes, and validation questions.

## Teaching patterns

### Safe Sandbox
- **Purpose**: Let player experiment with a new concept with low penalty.
- **Ingredients**: clear goal, low failure cost, constrained actions.
- **Common mistakes**: too many mechanics; unclear feedback; punitive reset.
- **Validation**: What does the first failure teach? Is retry < 2s?

### Isolated Mechanic
- **Purpose**: Introduce one mechanic with irrelevant complexity removed.
- **Ingredients**: single new idea, one clean interaction.
- **Common mistakes**: combining too early; decorative distractions.
- **Validation**: If you remove all other mechanics, does the level still work? (it should)

### Forced First Use
- **Purpose**: Make the only obvious first action use the new mechanic.
- **Ingredients**: constrained layout; affordance points to mechanic.
- **Common mistakes**: hidden first action; “gotcha” failure.
- **Validation**: What will 80% of players try first? Is that the intended experiment?

### Demonstration Before Demand
- **Purpose**: Show effect before requiring skillful use.
- **Ingredients**: safe demo moment; then a test.
- **Common mistakes**: demand without showing; effect not readable.
- **Validation**: Can a cold player predict the mechanic after seeing it once?

### Constraint Teaching
- **Purpose**: Teach by constraining possibilities, not by explaining.
- **Ingredients**: reduced action set; layout that makes correct experiment natural.
- **Common mistakes**: over-constraining to the point of boredom.
- **Validation**: Are irrelevant actions removed? Is the core action still a choice?

## Puzzle patterns

### Double Take
- **Purpose**: Make an obvious first solution fail readably, forcing recontextualization.
- **Ingredients**: clear goal; plausible first attempt; readable failure; low retry cost; second solution reuses same elements smarter.
- **Common mistakes**: failure feels unfair; correct solution requires hidden info; failure teaches nothing.
- **Validation questions**:
  - What will most players try first?
  - Why does it fail?
  - What does the failure teach?
  - What assumption must change?
  - Is the second solution visible after the first failure?

### Obvious Wrong Solution
- **Purpose**: A tempting but incomplete approach teaches a constraint.
- **Ingredients**: decoy route; clear failure reason; feedback.
- **Common mistakes**: decoy is random; failure ambiguous.
- **Validation**: Does wrong path teach exactly one new constraint?

### Setup and Payoff
- **Purpose**: Prepare a state/object first, then use it later.
- **Ingredients**: preparation area; payoff area; clear linkage.
- **Common mistakes**: payoff not linked; preparation too hidden.
- **Validation**: Is the payoff predictable from the setup?

### Lock and Key
- **Purpose**: Gate progress until a required object/state/skill is acquired.
- **Ingredients**: clear lock; reachable key; feedback on lock.
- **Common mistakes**: key not discoverable; lock looks decorative.
- **Validation**: Can player identify lock before key? Is key acquisition teachable?

### Multi-Step Causal Chain
- **Purpose**: Solution requires understanding linked effects in sequence.
- **Ingredients**: 2–4 causal steps; readable feedback per step.
- **Common mistakes**: chain too long; hidden intermediate state.
- **Validation**: Can player explain step N after seeing step N-1 fail?

### Recontextualization
- **Purpose**: Familiar space/object gains new meaning after new mechanic/state.
- **Ingredients**: same space reused; new rule changes interpretation.
- **Common mistakes**: novelty is cosmetic only.
- **Validation**: What changed in the player’s mental model?
